## 1) บริบทพริกปัจจุบัน

- **พริก (Prik)** = ผู้ช่วย AI ในฟลีตของเฮีย — เป็นน้องร่วมฟลีตกับ **พอส (Pors)** ซึ่งประจำเครื่อง
  **Mac mini M4 Pro 24GB** (เฮีย setup ให้ Claude ทำงานผ่านเครื่องนี้)
- **หน้าที่หลักของพริก: ดูแลอีเมล 5 บัญชี** — [ต้องยืนยัน: รายชื่อบัญชีทั้ง 5 จาก config จริง
  ก่อนแตะเมลใด ๆ] และจำไว้ว่าในจำนวนเมลทั้งหมด **เมลของ mink และ chalit ห้ามแตะเด็ดขาด** (ดูข้อ 4)
- พริกมี **regression suite** ของตัวเอง — [ต้องยืนยัน: path + คำสั่งรัน บนเครื่องจริง]
- พริกเชื่อมระบบผ่าน **bridge** — [ต้องยืนยัน: bridge ตัวไหน อยู่ที่ไหน สถานะปัจจุบัน]
- โมเดล/รันไทม์ที่พริกใช้ — [ต้องยืนยันจากเครื่องจริง]

**กติกาเมื่อเจอ [ต้องยืนยัน]:** หาคำตอบจากไฟล์/config บนเครื่องก่อน ถ้าไม่เจอให้ถามเฮีย
ห้ามสรุปเอาเองแล้วลงมือ

## 2) สิ่งที่เพิ่งทำกับพริกไปแล้ว — ห้ามทำซ้ำ / ห้าม revert

| งาน | สถานะ | หมายเหตุ |
|---|---|---|
| แก้สิทธิ์ **fleet relay** | ✅ เสร็จแล้ว | อย่าแก้ซ้ำ อย่า revert ถ้าเจอของที่ดูแปลกให้ถามเฮียก่อน |
| **roster patch** | ✅ เสร็จแล้ว | เช่นกัน — patch ไปแล้ว อย่า patch ทับ |
| โน้ต **เลขศาสตร์ 552** | ✅ แชร์ถึงผู้รับแล้ว | **ไม่ต้องส่งซ้ำ** เด็ดขาด (กันสแปมผู้รับ) |

รายละเอียดเชิงลึกของงานพวกนี้อยู่กับ session ที่ทำ — ถ้างานใหม่ต้องไปยุ่งพื้นที่เดียวกัน
ให้เช็คสถานะปัจจุบันของจริงก่อน แล้วแจ้งเฮียว่าจะแตะอะไร

## 3) สูตรฝึกที่พิสูจน์แล้วกับพอส — ใช้ลำดับนี้เป๊ะ ห้ามข้ามขั้น

```
1. Baseline    — วัดพฤติกรรมปัจจุบันก่อนแก้อะไร เก็บผลไว้เป็นตัวเทียบ
2. แก้ที่ราก    — วิเคราะห์ root cause แล้วแก้ที่ต้นเหตุ ไม่ patch กลบอาการ
3. Regression = 0 — รัน regression suite เดิมทั้งชุด ต้องผ่าน 100% (fail = 0) ก่อนไปขั้นถัดไป
4. Adversarial — ทดสอบเคสเจตนาหลอก/มุมร้าย/ขอบเขตแปลก ๆ
5. Live test   — ทดสอบกับของจริงแบบคุมความเสี่ยง แล้วค่อยถือว่าผ่าน
```

- ทุกขั้นต้องมีหลักฐานผลลัพธ์จริง (log/ผลรัน) ไม่ใช่ "น่าจะผ่าน"
- **Grounding gate ของพอส** (กลไกกันมโน — บังคับให้ตอบจากหลักฐานเท่านั้น ไม่มีหลักฐาน = บอกว่าไม่รู้)
  พิสูจน์แล้วว่าได้ผลกับพอส → **ควรพอร์ตไปใส่พริกด้วย** ถือเป็นงานหลักชิ้นหนึ่งของการฝึกรอบนี้
  [ต้องยืนยัน: ตำแหน่ง implementation ของ gate ฝั่งพอส เพื่อใช้เป็นต้นแบบ]

## 4) ข้อห้ามเด็ดขาด 🚫

1. **เมลของ mink และ chalit — ห้ามแตะทุกกรณี** ไม่อ่าน ไม่ย้าย ไม่ label ไม่ลบ ไม่ตอบ
2. **Backup ก่อนแก้เสมอ** — ไฟล์/config/ข้อมูลใด ๆ ต้องมีสำเนาก่อนแตะ และรู้วิธี restore
3. **ระวัง 2 session ทำงานพร้อมกัน** — ไฟล์ shared อาจถูกอีก session แก้อยู่:
   - ก่อนแก้ไฟล์ shared ให้เช็ค timestamp/สถานะล่าสุดก่อนเสมอ
   - อย่าเขียนทับโดยไม่ดูว่ามีการเปลี่ยนแปลงใหม่กว่าที่ตัวเองรู้
   - งานยาว ๆ ให้แจ้งเฮียว่ากำลังถือไฟล์ไหนอยู่
4. **ห้ามทำระบบพัง** — ถ้าไม่มั่นใจผลกระทบ ให้หยุดแล้วถาม อย่าเสี่ยง

## 5) สไตล์การทำงานที่เฮียคาดหวัง

- **พูดความจริงเท่านั้น** — ไม่รู้ = บอกว่าไม่รู้, ทำพลาด = รายงานตรง ๆ ทันที, ไม่แต่งผลลัพธ์
- **ทีละขั้น** — เสนอแผน → ยืนยัน → ลงมือ → รายงานผล แล้วค่อยขั้นถัดไป ไม่รวบยอด
- **ไม่พังเด็ดขาด** — ความเสถียรสำคัญกว่าความเร็ว งานช้าแต่ชัวร์ดีกว่าเร็วแล้วพัง
